# Pull Requests

Create GitHub Pull Requests with clear English titles and concise descriptions
that help reviewers understand what changed, why it changed, and how it was
verified.

Use GitHub CLI when the user asks to create a Pull Request from the local
repository, when a GitHub remote is available, and when `gh` is authenticated.

## Title

Use the Conventional Commits subject format for the Pull Request title:

```text
<type>(<scope>): <subject>
```

The scope is optional.

## Types

| Type | When to use |
|---|---|
| `feat` | New feature |
| `fix` | Bug fix |
| `docs` | Documentation only |
| `style` | Formatting, no logic change |
| `refactor` | Neither a fix nor a feature |
| `perf` | Performance improvement |
| `test` | Add or update tests |
| `chore` | Maintenance, dependencies |
| `build` | Build system changes |
| `ci` | CI configuration |
| `revert` | Revert a previous commit |

## Title Rules

**Subject**: mandatory, imperative mood, max 50 characters, lowercase after the
colon, no trailing period.

**Scope**: optional, area of the codebase in parentheses, such as `auth`, `api`,
`ui`, `db`, `config`, `docs`, or `tests`.

Use a title that describes the Pull Request as one coherent change.

## Description Format

Use this format:

```markdown
## Summary

- <change 1>
- <change 2>
- <change 3>

## Why

<brief explanation of the context or motivation>

## Verification

- <test, command, or manual check>
- <test, command, or manual check>

## Notes

<optional reviewer notes, risks, follow-ups, or migration details>
```

## Description Rules

Write the Pull Request description in English.

Keep the summary focused on what changed. Explain why the change is needed when
the context is not obvious. Include verification steps whenever possible.

If tests were not run, state that explicitly in the verification section.

Use the notes section only when it adds useful reviewer context, such as risks,
follow-ups, migration details, or compatibility concerns.

## Breaking Changes And Issues

Breaking change:

```text
BREAKING CHANGE: <description>
```

Issue reference:

```text
Closes #123
Refs #456
```

## Before Creating The Pull Request

Review the diff before writing the Pull Request.

The Pull Request must represent one coherent change. If the branch includes
unrelated work, say that the branch should be split before opening the Pull
Request.

Check the current branch, base branch, remote, and authentication state before
running GitHub CLI commands.

Useful checks:

```bash
git status --short
git branch --show-current
git remote -v
gh auth status
```

## GitHub CLI

Create the Pull Request with `gh pr create`.

Prefer explicit title and body values instead of relying only on autogenerated
content:

```bash
gh pr create --base <base-branch> --head <head-branch> --title "<title>" --body "<body>"
```

Use `--draft` when the Pull Request is not ready for review:

```bash
gh pr create --base <base-branch> --head <head-branch> --draft --title "<title>" --body "<body>"
```

Use `--fill` only when the commit history already contains a clear title and
description:

```bash
gh pr create --base <base-branch> --head <head-branch> --fill
```

Use `--web` only when the user explicitly wants to finish the Pull Request in
the browser:

```bash
gh pr create --web
```

## Workflow

1. Inspect the working tree and confirm there are no unexpected changes.
2. Identify the current branch and the intended base branch.
3. Review the diff and commit history.
4. Draft a Conventional Commit style Pull Request title.
5. Draft a concise Pull Request description using the required format.
6. Include verification steps, or state that tests were not run.
7. Create the Pull Request with `gh pr create` when requested and safe to do so.
8. Report the Pull Request URL after creation.

## Examples

```text
feat(auth): add OAuth2 login support
```

```markdown
## Summary

- Add OAuth2 login support for Google and GitHub
- Wire provider callbacks into the authentication flow
- Update authentication configuration

## Why

Users need a secure external login option without creating a local password.

## Verification

- npm test
- Manually verified Google login in the local environment

## Notes

Closes #142
```

```text
fix(api): handle null user responses
```

```markdown
## Summary

- Return 404 when the user response is null
- Avoid throwing an internal server error during authentication failures

## Why

Missing users should be handled as expected client-facing failures, not server
errors.

## Verification

- npm test
```
